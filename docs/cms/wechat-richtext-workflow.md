# 统一富文本与微信公众号复制流程

## 交付目标

过渡期允许新媒体人员在 TypeZen 编辑一次，再把同一份富文本分别粘贴到微信公众号和 CMS；CMS 会识别 TypeZen 剪贴板结构、归档 base64 图片并保存受限原样导入块。目标流程仍是在 CMS 编辑一次，官网详情页与“复制到公众号”读取同一个服务端发布快照。

本期不保存微信公众号 AppID、AppSecret，也不调用公众号草稿或发布 API。CMS 负责生成兼容富文本并写入剪贴板，最终发布仍由工作人员在微信公众号后台确认。

## 实际工作流

1. 在 CMS 中填写标题、摘要、分类、封面和正文。
2. 正文编辑器支持三级小标题、加粗、斜体、下划线、删除线、字号、字体、前景色、背景色、对齐、列表、引用、链接、代码、分隔线、表格和图片。
3. 从 TypeZen 粘贴整篇图文时，CMS 不交给 Tiptap 默认 HTML 解析，而是作为 TypeZen 原样导入块处理；base64 图片先上传到 CMS，危险 HTML 和 CSS 在服务端净化。
   TypeZen 原样导入块不参与 CMS 自动翻译，三语稿需分别导入对应语言排版；需要继续使用自动翻译时，应使用 CMS 原生正文。
4. 图片选择、拖入或粘贴后先上传到 CMS。透明图保存为 PNG，其他图片保存为 JPEG。
5. 保存草稿时，服务端从统一文档重新生成白名单内联样式 HTML，并写入 `publicationHtml`。
6. 官网预览和正式详情页直接使用保存后的 `publicationHtml`。
7. 点击“一键复制到公众号”时，CMS 同时写入剪贴板的 `text/html` 和 `text/plain`。图片优先转换成剪贴板可携带的数据；超出大小或读取失败时保留稳定的绝对 HTTPS 地址。
8. 在微信公众号编辑器正文区粘贴，检查微信预览后发布。

## 内容模型

每个 `cn/jp/hk` 语言版本使用相同结构：

| 字段 | 作用 |
| --- | --- |
| `body.schemaVersion` | 当前固定为 `cms-richtext.v1` |
| `body.editorDocument` | 唯一可编辑源，采用受限的 Tiptap JSON 节点和标记 |
| `body.styleConfig` | 模板、主题色、字号、行高、段距、留白和图片圆角 |
| `body.assets` | 正文图片的 CMS 素材元数据 |
| `body.publicationHtml` | 服务端生成的官网/公众号共用 HTML 快照 |
| `body.contentHash` | 发布快照 SHA-256，用于一致性检查 |
| `body.renderVersion` | 当前渲染器版本 `wechat-inline.v2` |

允许的正文节点和属性在 `lib/cms/rich-text.ts` 统一净化；输出 HTML 在 `lib/cms/publication-renderer.ts` 统一转义、URL 校验和内联样式渲染。若渲染器升级，必须重新执行富文本迁移器刷新所有历史版本快照。

## 浏览器与微信边界

- 推荐使用最新版 Chrome 或 Edge，并通过 HTTPS 或本机 `localhost` 访问 CMS；现代剪贴板 API 需要安全上下文和用户点击触发。
- 微信公众号编辑器会再次过滤 HTML。CMS 只使用常见标签和内联样式，但工作人员仍必须以微信后台的手机预览作为最终验收。
- “已写入剪贴板”不等于“已通过微信公众号兼容验收”；标准样稿必须实际粘贴到公众号后台检查。
- 本机 `/uploads/...` 图片可以在本地复制测试；生产使用必须配置 Vercel Blob，使兜底图片地址为公网 HTTPS。
- 单张图片超过 2 MB，或一次复制的嵌入图片累计超过 8 MB 时，剪贴板保留公网地址，不强行写入大体积数据。
- 每次成功复制会记录 `COPY_WECHAT_HTML` 审计动作、语言、模板和公网图片兜底数量。

## 不在本期范围

- 直接创建或发布微信公众号草稿。
- 微信永久素材、临时素材和 access token 管理。
- 复制后自动绕过微信自己的内容过滤。
- 从微信公众号后台反向同步排版。
