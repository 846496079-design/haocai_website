# CMS 非阻塞翻译发布实施计划

## 目标

允许中文稿在未翻译时正常保存、预览和发布，同时保证 JP/HK 只展示当前发布版本中已准备好的外语内容，并补齐自有服务器 API Key 配置和验收说明。

## 实施步骤

### 1. 拆分内容完整度

- 在 `lib/cms/types.ts` 保留现有三语完整度函数，新增单语言完整度函数。
- 发布门禁只校验 CN；外语公开状态继续使用三语完整度和译稿状态共同判断。

### 2. 固化发布版本的外语可见性

- 在 SQLite 和 PostgreSQL 的 `news_article` 增加 `published_locales_complete`。
- 增加独立 PostgreSQL 迁移，使用空值回填避免后执行迁移覆盖新发布的明确状态。
- 既有已发布稿按旧发布门禁事实回填为外语可见；新稿在每次发布事务中重新计算。
- SQLite 初始化、PostgreSQL 运行时兼容和历史 seed 使用同一字段语义。

### 3. 调整公开读取

- CN 继续返回全部已发布稿。
- JP/HK 查询只返回 `published_locales_complete` 为真的已发布稿。
- 保留既有排序、置顶、详情查找和运行时动态读取逻辑。

### 4. 调整 CMS 编辑器

- 发布按钮只受中文完整度、保存状态、上传状态和当前版本预览约束。
- 将发布检查和人工确认文案改为中文可发布、外语待翻译的真实状态。
- 保留一键翻译、补全/覆盖模式、预览、复制、下线和审计功能。

### 5. 同步配置与运维依据

- 更新翻译 API、环境变量、自有服务器和部署验收文档。
- 写明共享 `.env.local` 的准确路径、`0600` 权限、PM2 重启和健康接口检查。
- 强调浏览器、仓库、CNB 参数和 `NEXT_PUBLIC_` 变量中都不得出现 API Key。

### 6. 自动验证

- 增加静态验证脚本，检查双存储发布门禁、JP/HK 过滤、迁移、编辑器按钮和服务端密钥边界。
- 执行 CMS 类型检查、迁移 dry-run、相关 CMS 验证、生产构建和 `git diff --check`。
- 使用隔离的临时 SQLite 数据库和空闲本机端口验证 CN 始终可见、JP/HK 按发布快照隐藏和恢复；脚本结束后自动停止服务并清理临时目录。

## 回滚

- 代码回滚后新增字段保留但不再参与读取，不删除数据库列或历史内容。
- 未翻译稿若已发布，回滚旧代码前应先完成翻译或下线，否则旧门禁只会阻止后续发布，不会自动撤销当前中文版本。
- API Key 配置错误时恢复旧值并重启 `zds-website`；不得通过前端临时传递密钥绕过服务端配置。
