# CMS 操作体验与静态交付设计

## 目标

修复 CMS 登录首请求延迟、已发布稿件置顶反馈、筛选栏对齐与新建草稿的路径录入问题；同时为技术交接提供公开站的静态 HTML 快照导出方式。

## 已确认的根因与处理

1. Neon 冷连接时，CMS 每次进程启动都串行执行多条建表语句，导致登录首请求明显变慢。已改为先用一条 `to_regclass` 查询确认已迁移 schema，存在时直接复用；空数据库仍保留完整建表兜底。
2. 置顶请求此前可执行，但界面只在完成后显示通用提示，缺少进行中与成功状态。已增加“正在置顶”及明确结果提示，保留仅已发布稿件可置顶的服务端校验。
3. 搜索框使用隐藏标签，其他筛选项使用可见标签，造成控件基线不一致。已统一为可见标签、相同纵向间距和底部对齐的筛选网格。
4. 文章路径是内部标识，不应由编辑手填。手工新建草稿由服务端分配 `draft-<文章ID>`；外部导入由稳定 `sourceId` 派生 `workspace-<哈希>`，用于同稿更新。

## 对接边界

- 新媒体工作台只传 `sourceId` 和 `content.cn`，不传 `slug`。
- 同一 `sourceId` 必须稳定，CMS 以此更新草稿；不同 `sourceId` 创建新稿件。
- 分类为空时允许导入为草稿并提示补充；新分类会自动创建，来源标记为外部导入。
- 置顶和人工排序只影响已发布稿件，接口仍以服务端状态校验为准。

## 静态 HTML 交付

运行 `npm run build` 后，以可访问公开站的 Next 服务为源执行 `npm run export:public-html`，会在 `dist/public-static` 生成 CN、HK、JP 的公开页面 HTML、`public` 资源和 Next 静态资源。该目录被 Git 忽略，作为交付构建产物，不进入源码仓库。

CMS 不能改为纯静态页面：其登录、Neon 数据库、图片上传和编辑 API 必须由服务端运行。公开静态快照也不构成源码保密措施；浏览器交付的 HTML、CSS 和客户端 JavaScript 本来就可被查看。真正的保密边界是：密钥只放服务端环境变量，不提交 `.env.local`，且不把 CMS 服务端代码或数据库凭据部署到静态托管目录。

## 验收

1. 已迁移 Neon 环境首次登录不再执行整套建表 DDL。
2. 点击置顶立即显示操作状态，成功后明确提示，刷新后顺序保持。
3. 搜索、分类、标签、开始日期和结束日期的标签与输入框对齐。
4. 新建草稿无需填写路径，返回编辑页后可继续填写内容。
5. 导入接口文档不再要求 `slug`，并明确 `sourceId` 的更新语义。
6. 可生成独立的公开站静态 HTML 快照，不影响 Node 方式运行的 CMS。
