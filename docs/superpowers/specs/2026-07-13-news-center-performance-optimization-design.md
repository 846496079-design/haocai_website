# 新闻中心性能优化设计

## 背景与结论

当前新闻内容集中维护在 `lib/news-content.ts`，并通过 Next.js 静态导出生成新闻列表与详情页。新闻页本地 HTML 响应约 0.1 秒，当前慢感的主要来源不是缺少数据库，而是新闻封面资源过大、静态预览服务缺少压缩缓存，以及客户端组件携带了超出列表展示所需的新闻数据。

本轮只优化现有新闻中心的加载性能，保留 CN、JP、HK 三站的文章数量、日期、标题、摘要、正文、分类、标签、路由和页面交互，不建设新闻后台，不改变线索提交、其他页面或发布架构。

## 方案比较

### 方案 A：静态架构内做针对性优化（本轮采用）

- 将新闻封面生成适合网页展示的 WebP 资源，保留原图用于回退。
- 新闻列表只消费卡片需要的元数据，避免把新闻正文作为列表数据进入客户端。
- 保持现有静态导出、路由和分类筛选行为。

优点是改动范围小、性能收益直接、不会阻塞后续 CMS 建设。缺点是新闻发布仍需通过代码和构建流程完成。

### 方案 B：先把新闻迁移为 Markdown 或 JSON 内容库

可以降低编辑 TypeScript 的门槛，但新媒体部门仍然需要文件、Git 和部署流程，不能真正解决独立发布与审核问题。本轮不采用。

### 方案 C：立即建设 CMS、接口和数据库

能够解决新媒体部门的内容管理问题，但会同时涉及账号权限、草稿审核、多语言、素材存储、发布回滚和部署方式调整，范围明显大于性能修复。本轮不采用，作为下一阶段独立立项。

## 本轮优化范围

### 1. 新闻封面资源

- 为当前被新闻数据引用的 PNG 封面生成 WebP 版本，最长边不超过 1600 像素，保持原始宽高比。
- WebP 质量以肉眼无明显损失为前提，目标是单张封面控制在约 100 KB 至 350 KB；真实活动照片可适当放宽。
- 新资源统一放在 `public/images/news/optimized/`，不覆盖或删除原文件。
- `lib/news-content.ts` 只替换封面路径，不改文章事实内容。

### 2. 新闻列表数据边界

- 明确新闻列表卡片的数据结构，只包含 `slug`、日期、分类、标签、标题、摘要和封面。
- 列表页和首页新闻摘要只消费列表元数据；详情页继续消费完整正文。
- 保留当前三站本地化映射和既有文章路由，不改变用户可见内容。

### 3. 图片加载行为

- 新闻列表封面继续采用延迟加载和异步解码。
- 首屏第一张新闻封面允许提高加载优先级，其余图片保持延迟加载，避免同时争抢网络带宽。
- 保持现有 16:9 卡片裁切和详情页视觉比例。

## 明确不做

- 不新增数据库、管理后台、登录权限或新闻发布接口。
- 不修改新闻正文、标题、日期、分类、标签和三语翻译。
- 不调整新闻卡片视觉设计、分类筛选方式和详情页信息架构。
- 不删除原始图片，不改其他模块的图片与交互。
- 不以本地 Python 静态服务器的响应头作为正式生产部署配置。

## 后续新闻管理阶段

新媒体部门后续需要独立发布和管理新闻，因此下一阶段应建设内容管理能力，而不是只增加一张数据库表。至少包括：

- 新媒体账号与角色权限；
- 草稿、审核、发布、下线和版本回滚；
- CN、JP、HK 三语关联及缺失翻译状态；
- 封面和正文素材上传、压缩与复用；
- 标题、摘要、正文、分类、标签、发布时间、SEO 路径管理；
- 发布后触发官网更新，并保留操作记录。

在 CMS 建设前需决定发布模式：保留静态站并由 CMS 触发重新构建，或者迁移到支持服务端渲染与按需更新的部署环境。数据库本身不直接改善页面加载速度。

## 验证标准

1. `npm run build` 成功，CN、JP、HK 新闻列表与全部详情页继续静态生成。
2. 三站文章数量、slug、日期、分类、标题和详情正文与优化前一致。
3. 新闻列表引用优化后的 WebP，所有资源请求成功，原始 PNG 保留。
4. 优化后新闻封面资源总体积相比当前引用资源至少降低 70%。
5. 新闻列表 HTML 与客户端数据不再重复携带完整文章正文。
6. 在 390px、768px、1440px 检查新闻列表和一篇新闻详情，布局、筛选、链接和图片显示正常。
7. 对修改文件执行 UTF-8 与中文乱码检查，页面不出现内部开发措辞。

## 风险与回退

- WebP 兼容现代主流浏览器；原始 PNG 保留，可通过恢复 `cover` 路径立即回退。
- 新闻数据拆分只改变内部数据边界；若构建或路由校验失败，恢复原导入方式，不影响文章源数据。
- 本轮不触碰生产部署和数据库，不会改变现有发布链路。
