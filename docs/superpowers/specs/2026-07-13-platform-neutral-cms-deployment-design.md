# CMS 平台无关部署与技术交接设计

## 背景与目标

新闻 CMS 已使用 PostgreSQL 持久化数据，并通过服务端接口管理图片和翻译。部署位置尚未确定，可能使用 Vercel，也可能运行在公司服务器。本设计把计算、数据库、对象存储和外部 API 分开，使计算平台的选择不影响现有数据。

## 方案比较

### 方案一：全部使用 Vercel 托管

优点是部署与回滚最快、运维工作少；缺点是运行平台和图片存储均依赖 Vercel，企业网络与成本策略受平台约束。

### 方案二：自有服务器计算，继续使用 Neon 与 Vercel Blob（推荐）

Next.js 服务运行在公司服务器，数据库仍为 Neon，图片仍为 Vercel Blob。应用层无须改造，迁移风险最低，也保留以后迁回 Vercel 的能力。服务器必须能通过 HTTPS 访问 Neon、Vercel Blob 和选定的翻译 API。

### 方案三：全部私有化

Next.js、PostgreSQL 和对象存储均由公司维护。控制力最高，但当前图片上传实现需要新增 S3/MinIO 存储适配器，还需承担数据库高可用、备份恢复和对象存储运维。本阶段不实施。

## 推荐架构

浏览器只访问 Next.js 应用。Next.js 服务端访问 Neon、Vercel Blob 和自有翻译 API，所有密钥仅存在于服务端环境变量。CMS 会话存储在 PostgreSQL，因此单实例和多实例部署共享登录状态；应用服务器不保存业务数据，可按同一镜像水平扩容。

## 部署边界

- 计算层：Vercel 或 Linux 服务器，二选一。
- 数据层：继续使用已授权的 Neon PostgreSQL。
- 图片层：第一阶段继续使用 Vercel Blob；完全私有化时另立改造任务。
- 翻译层：保留自有 Chat Completions 兼容 API，未配置时仅禁用一键翻译，不影响稿件编辑。
- 域名与 TLS：由技术总确定生产域名、证书和 DNS 切换窗口。

## 数据与错误处理

- schema 迁移由单一操作者在发布窗口显式执行，不放进自动构建命令。
- 历史 seed 按 slug 幂等；有效记录不覆盖，指针缺失或无效时只修复版本关系。
- 每次发布前执行 `cms:verify`，检查 schema、版本指针、三语正文和历史内容哈希。
- 应用回滚与数据库恢复分开处理；仅前端或运行时问题不得覆盖数据库。

## 验收标准

1. 生产服务能连接 Neon 与 Blob，未退回 SQLite 或本地上传目录。
2. 17 篇历史已发布新闻通过 SQLite 与 Neon 对账。
3. 管理员可登录并完成分类、草稿、图片、翻译、预览、发布、下架和恢复流程。
4. CN、JP、HK 新闻列表、详情、置顶和排序一致。
5. 技术总掌握环境变量、迁移、备份、回滚、监控和密钥轮换责任。
