# CMS 统一富文本与微信公众号兼容排版设计

| 项目 | 内容 |
| --- | --- |
| 版本 | V1.1 |
| 日期 | 2026-07-17 |
| 状态 | 已完成 TypeZen 兼容修复，正在执行回归；微信后台终验前不得标记完全兼容 |
| 适用范围 | 新闻 CMS、三语稿件、官网预览与发布、微信公众号富文本复制 |
| 核心决策 | 单一结构化富文本模型；微信公众号式所见即所得编辑；官网与剪贴板复用同一发布快照 |

## 1. 背景与目标

当前 CMS 正文以 `lead / sections / closing` 为核心。该结构适合固定官网新闻模板，但无法完整表达自由标题层级、局部颜色、字体、引用、列表、表格、代码、连续图片与公众号式排版。新媒体人员需要逐块搬运内容，官网和微信公众号也容易形成两套排版结果。

当前过渡流程是工作人员在 TypeZen 完成一次排版，把同一份富文本分别粘贴到微信公众号和 CMS。目标流程是直接在 CMS 排版，再一键复制到微信公众号。两种流程必须使用同一套可验证的微信公众号 HTML 兼容边界，不能仅以浏览器成功写入剪贴板作为兼容证明。

目标：

- 采用微信公众号式所见即所得编辑器，工作人员不需要理解 HTML 或 Markdown。
- 结构化富文本是唯一可编辑正文模型，旧正文模型完成迁移后退出运行链路。
- 官网预览、官网正式发布与微信公众号复制共享同一份经过净化的内联样式 HTML 快照。
- 保留三语、草稿、预览、人工审核、发布、下架、回收站、翻译与审计能力。
- 标准样稿粘贴到微信公众号后不需要逐段重新排版。
- TypeZen 标准样稿粘贴到 CMS 后，文字、层级、颜色、布局和图片顺序不因 Tiptap 默认解析而丢失。
- CMS 提供可配置的 AI 智能排版建议，模型不得改变原文、链接或图片，采用前必须人工对照。

本期不接入 AppID、AppSecret，不直接调用微信公众号草稿或正式发布接口。

## 2. 内容模型

标题、作者、摘要、发布日期、分类、标签和封面继续作为独立业务字段。正文统一为结构化富文本。

每种语言保存：

| 字段 | 责任 |
| --- | --- |
| `editorDocument` | 唯一可编辑源，保存有序节点、文字与格式标记 |
| `styleConfig` | 模板、主题色、正文参数和图片参数 |
| `assets` | 正文素材引用及其稳定地址、尺寸、哈希、替代文字和说明 |
| `publicationHtml` | 服务端根据源稿生成并净化的内联样式 HTML 快照 |
| `contentHash` | 当前发布快照的 SHA-256，用于一致性与预览门禁 |
| `renderVersion` | 渲染规则版本，用于回归与重建 |

`publicationHtml` 是派生发布结果，不是第二套人工编辑模型。浏览器不能直接把任意 HTML 写入数据库。

`editorDocument` 允许一个受限的 `wechatHtmlBlock` 原子节点，作为 TypeZen 过渡期无损导入边界。该节点仍属于统一文档，不另建人工维护的第二套正文；导入 HTML 必须在浏览器初筛、图片归档后，再由服务端执行标签、属性、URL 和 CSS 白名单净化。原样导入块在 CMS 中整体预览、删除或重新导入，不承诺逐元素反向编辑，也不参与自动翻译，三语稿必须分别导入对应语言；CMS 原生创建的正文继续使用结构化节点编辑和自动翻译。

结构化正文支持：

- 段落、三级正文标题、硬换行和分隔线。
- 加粗、斜体、下划线、删除线、文字色、背景色。
- 左对齐、居中、右对齐。
- 有序列表、无序列表、引用、链接。
- 行内代码、代码块和简单表格。
- 图片、替代文字、图片说明和连续图片。

字体使用经验证的系统字体白名单，不保存外部网络字体。

## 3. 编辑器体验

CMS 保留现有基础信息、三语切换、翻译、预览和发布区域，正文区域替换为所见即所得编辑器。

- 固定工具栏提供段落样式、文字格式、颜色、对齐、列表、链接、图片、表格、撤销与重做。
- 编辑区直接显示接近微信公众号的排版效果。
- 右侧提供模板、主题色、字号、行高、段间距、页面留白和图片圆角。
- 手机宽度预览与编辑器使用同一渲染结果。
- 输入内容有明确的未保存、保存中、保存失败和已保存状态。
- 复制、预览和发布只允许使用已保存且与当前内容一致的快照。
- 键盘操作、焦点状态、禁用状态和错误提示满足后台工具的无障碍要求。

首期提供 6 套原创模板：品牌科技、商务简约、新闻通告、产品发布、行业洞察和活动宣传。模板不复制参考仓库的代码、品牌或视觉资产。

## 4. 确定性渲染与安全净化

保存草稿时由服务端执行：

1. 校验富文本节点、属性、文字标记和素材引用。
2. 根据模板和排版参数生成确定性 HTML。
3. 把允许的样式写入元素 `style` 属性，不依赖外部 CSS 或类名。
4. 删除危险标签、事件属性、不安全 URL 和不兼容样式。
5. 生成 `publicationHtml`、`contentHash` 和 `renderVersion`。
6. 在同一版本写入中保存编辑源与发布快照。

允许的 HTML 以 `section`、`p`、`span`、`strong`、`em`、`u`、`s`、`a`、`img`、`blockquote`、`ul`、`ol`、`li`、`hr`、`br`、`code`、`pre` 和简单表格标签为主。

禁止脚本、`iframe`、表单、事件属性、任意外部 CSS、固定定位、动画、复杂 Grid/Flex、伪元素关键文字和未经校验的原始 HTML。

CMS 原生渲染采用微信公众号稳定结构：标题使用嵌套 `section`，列表使用清除浮动的分栏结构，连续图片使用 `inline-block`，表格同时写入内联背景色和 `bgcolor` 兜底。不能把标准浏览器中的 `h2`、`ul`、Flex 或外部样式可用，等同于微信公众号粘贴后可用。

## 5. 图片与剪贴板

粘贴截图、拖入图片和文件选择统一走 CMS 素材服务：

1. 在当前光标位置创建上传占位节点。
2. 服务端校验图片类型、尺寸与体积。
3. 照片转换为 JPEG，需要透明背景时保留 PNG。
4. 生成稳定 HTTPS 地址、尺寸、内容哈希和素材 ID。
5. 上传成功后原位替换占位节点；失败时恢复原编辑位置。

上传中的图片阻止保存、复制和发布。正文不保存 `blob:` 临时地址，也不把 Base64 作为长期内容。

执行“一键复制到公众号”时：

- 校验草稿已保存、快照未过期、图片完整且地址有效。
- 将相对图片地址转为绝对地址。
- 对可安全读取且体积合适的图片生成剪贴板传输表示；失败时保留稳定公网地址并提示。
- 同时写入 `text/html` 和 `text/plain`。
- 新版 Clipboard API 不可用时，降级为隐藏富文本选区复制。
- 记录文章版本、模板、复制时间和结果。

剪贴板传输表示只服务于粘贴，不形成第二套编辑模型。

## 6. 三语与翻译

中文仍是唯一源稿。CN、JP、HK 使用相同富文本结构；翻译只替换文字节点，保留节点 ID、标题层级、图片、链接和排版标记。

- 中文变化后，日文和繁体状态变为“源稿已更新”。
- 重新翻译支持覆盖机器译文，或保留人工内容并补全变化节点。
- 模板、主题色与布局默认三语共享；字体栈根据语言调整。
- 单一语言生成失败不破坏其他语言和线上版本。
- 正式发布继续要求三语完整并完成当前版本预览。

## 7. 旧模型迁移

迁移映射：

- `lead` 转为导语段落。
- 每个 `sections[].title` 转为标题节点。
- `sections[].paragraphs` 按原顺序转为段落节点。
- 分节图片、替代文字与图片说明转为图片节点。
- `closing` 转为结语段落。
- 标题、摘要、封面、日期、分类和标签继续作为独立字段。

迁移要求：

- 迁移前备份数据库并生成文章、版本和语言清单。
- 迁移脚本幂等；重复执行不产生重复节点或素材。
- 自动比较文字顺序、标题、图片数量和图片顺序。
- 任一语言失败时不切换该文章，不允许半迁移。
- 先生成并验证新版数据，再切换官网读取。
- 切换期间只保留可回滚保护层，不长期维护双写。
- 人工验收和观察完成后删除旧编辑器、旧渲染器和旧模型运行代码；旧数据仅存在于迁移备份。

## 8. 草稿、预览与发布

- 浏览器保留本地恢复稿；服务端防抖自动保存工作副本。
- 明确保存草稿时生成版本号、HTML 快照和哈希。
- 编辑已发布文章生成独立草稿版本，不直接修改线上版本。
- 预览直接读取保存后的 `publicationHtml`，不在页面端重新解释正文。
- 只有当前内容哈希与已预览哈希一致时才能发布。
- 发布以数据库事务切换正式版本；失败时线上版本保持不变。
- 公众号复制与官网发布互不自动触发，也不强制绑定发布时间。

## 9. 异常与审计

- 网络中断保留本地恢复稿并显示未同步状态。
- 版本冲突停止覆盖并提示重新载入或复制当前内容。
- 图片失败恢复原位置，不留下残缺节点。
- HTML 生成失败保留编辑源，但阻止预览、复制和发布。
- 不兼容格式指出具体位置，不静默丢失文字。
- 翻译失败保留中文、已有译文和线上版本。
- 剪贴板失败提示权限或启用降级复制。
- 发布或缓存刷新失败回滚版本切换。

审计日志覆盖保存、翻译、预览、复制、发布、下架、模板变更、迁移、回滚和版本冲突。

AI 服务配置只保存在当前浏览器，调用时临时传给 CMS 服务端转发，不写入数据库和审计明文。服务端必须拒绝本机、内网、DNS 解析到私网和非 HTTPS API 地址，同时禁止跟随重定向；返回后比较原稿与建议稿的全部文字、链接和图片，任一项变化即拒绝建议。

## 10. 验收标准

自动化验证：

- 富文本节点、样式、素材与 URL 校验。
- 相同输入生成相同 HTML 和哈希。
- 标签、属性和样式白名单及 XSS 用例。
- 旧文章迁移顺序、文字、图片和幂等测试。
- 保存、重新打开、翻译、预览、复制、发布、下架和恢复流程。
- CN、JP、HK 三站公开页面回归。
- Chrome、Edge 富文本剪贴板与降级复制。
- TypeZen 源码渲染样稿经过剪贴板导入后，标题、列表、高亮、链接和图片不再退化为普通段落或被删除。
- TypeZen 导入 HTML 中的脚本、事件属性、危险 URL 和非白名单 CSS 被服务端删除。
- AI 建议改变任一文字、链接或图片时由服务端拒绝。
- 简体、繁体、日文 UTF-8 乱码检查。

真实微信公众号标准样稿覆盖三级标题、颜色、安全字体、引用、列表、链接、单图、多图、长图、透明图片、图片说明、分隔线、表格、代码和中日文混排。

通过条件：

- 官网与公众号的文字、顺序、图片数量及顺序一致。
- 标题层级、字号、颜色、间距和对齐不需要重新排版。
- 图片没有丢失、错位或临时地址。
- 微信不支持的字体只回退为约定系统字体，不影响层级和可读性。
- 标准样稿不需要逐段修复。
- 旧文章没有丢字、乱码、图片缺失和公开链接变化。
- 公共网站不加载编辑器依赖，性能不因 CMS 编辑器退化。

## 11. 上线顺序

1. 增加新内容类型、渲染器、编辑器和迁移工具，不切换线上读取。
2. 在本地及预发布环境迁移全部文章并执行自动比较。
3. 人工抽查 CN、JP、HK，完成真实公众号粘贴验收。
4. 切换 CMS 正式写入和官网读取到新模型。
5. 保留可回滚备份并观察线上结果。
6. 稳定后删除旧模型运行代码与临时兼容开关。

## 12. 2026-07-17 实施同步与纠偏

首次实施只验证了 CMS 成功写入 `text/html` 和 `text/plain`，没有把 TypeZen 输出实际粘入 CMS 做结构对照，也没有完成微信公众号后台粘贴终验，因此“完全兼容”的结论不成立。源码对照和同一样稿回归确认：TypeZen 输出依赖特定的嵌套 `section`、浮动列表、连续图片、内联样式和 base64 图片；旧 CMS 会把标题和列表拆成普通段落、丢失高亮与模板样式，并因 `allowBase64: false` 删除图片。

纠偏后已落地：

- CMS 正文持久化已切换为 `cms-richtext.v1`，`editorDocument` 是唯一可编辑源；浏览器提交的 HTML 会被忽略，SQLite 和 PostgreSQL 保存前都由服务端重新净化、渲染和计算哈希。
- CMS 编辑器保留结构化所见即所得能力，并新增 TypeZen 剪贴板识别。TypeZen 图文作为受限原样导入块进入统一文档，base64 图片在插入前上传到 CMS，在线图片会明确提示长期可用性风险。
- 官网预览与公开详情已读取同一个 `publicationHtml`；公共页面没有导入 Tiptap 编辑器运行时。
- 微信渲染器升级为 `wechat-inline.v2`；标题、列表、连续图片和表格改用微信公众号更稳定的结构。“一键复制到公众号”继续同时写入 `text/html` 与 `text/plain`，优先携带图片数据，失败或超限时使用绝对公网地址。
- CMS 已增加 OpenRouter、OpenAI 兼容接口和 Anthropic 配置入口。AI 只返回结构建议，服务端验证原文、链接、图片完全一致后才允许人工采用。
- 上传链路已改为透明图 PNG、其他图片 JPEG，避免把 WebP 作为公众号复制的唯一图片格式。
- 翻译链路只替换有序文字节点，保留节点结构、样式和图片。
- 草稿保存已增加版本号与更新时间并发校验，检测到其他窗口写入时返回冲突，不再静默覆盖。
- 新媒体导入接口已改为 `cms-richtext.v1` 契约；旧结构仅在导入和迁移边界一次性转换，不会继续写入数据库。
- 已增加 SQLite/PostgreSQL 共用的幂等迁移器、备份门禁、文字/图片顺序比较和当前渲染快照校验。

- `npm run cms:verify-richtext` 结果为 19 个版本、待迁移 0 个。

首次实施验证记录保留用于追溯，但不能作为兼容结论：

- `npm run typecheck:cms` 通过。

- 渲染器白名单用例确认脚本标签、`javascript:` 链接和非法图片地址不会进入发布 HTML。
- 本地浏览器确认：已发布稿件载入后保持“已保存”，编辑器工具栏可用，手机预览显示迁移后的标题、段落和图片，不会因 Tiptap 默认属性触发误保存。

本地兼容回归已完成：从 TypeZen 在线编辑器读取同一标准样稿的实际渲染 DOM，粘贴进 CMS 后保存并重开；CMS 再次复制时保留标题、强调、浮动列表和并排多图结构，并把 4 张远程图归档为内嵌图片。该结果证明 TypeZen 到 CMS 再到富文本剪贴板的链路一致，但不替代微信公众号后台的最终过滤验收。
