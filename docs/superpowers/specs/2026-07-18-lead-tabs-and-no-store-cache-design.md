# 官网线索分区与公开页面无缓存设计

## 1. 背景与根因

### 1.1 产品对比方案仍显示旧内容

生产服务器当前已经返回新版产品对比方案，包括“费用方式”“适用纳税人类型”和“工作方式与产品优势”。但曾经访问过旧版 `/cn/product/` 的浏览器仍可能直接复用历史一年缓存，并加载旧版 JavaScript，因此继续显示“年度费用”等旧表格。

当前 `no-cache` 只会约束浏览器取得新响应后的复用行为，无法远程删除已经写入用户设备的一年缓存。带新查询参数访问时会绕过旧缓存并取得当前版本，证明生产 release 和页面数据源本身已经更新。

### 1.2 CMS 线索类型不够清晰

线索数据库和接口已经用 `TRIAL`、`PARTNER` 区分体验用户线索与代理商合作线索，列表也支持类型筛选。但当前 CMS 只在同一列表中提供下拉筛选和一行小字标签，没有形成明显的业务工作区，也没有按类型展示数量，因此管理员感知上仍是“所有线索混在一起”。

## 2. 目标

1. 公开 HTML 与 RSC 响应不在浏览器或共享缓存中保存，每次请求都向当前生产服务获取响应。
2. 保留带内容哈希的 JavaScript、CSS、字体和 Next.js 静态资源长期缓存，避免无必要的重复下载。
3. CMS 线索管理页提供“全部线索”“用户线索”“代理商线索”三个明显页签。
4. 每个页签展示对应线索数量，并与状态、超时筛选协同工作。
5. 不迁移或复制现有线索数据，不改变公开表单、异步发送、重试和加密存储逻辑。

## 3. 非目标

1. 不建设销售分配、跟进记录、商机阶段等 CRM 功能。
2. 不把两类线索拆成两张数据库表或两套详情页面。
3. 不禁用带哈希静态资源缓存。
4. 不承诺服务器能主动删除用户设备中已经存在的历史一年缓存。
5. 不修改新闻内容管理和 CMS 富文本发布流程。

## 4. 缓存策略设计

### 4.1 公开 HTML 与 RSC

Nginx 对 `/cn/`、`/jp/`、`/hk/` 及其子页面统一覆盖为：

```text
Cache-Control: no-store, max-age=0
```

HTML 和 `text/x-component` RSC 响应必须使用同一规则。`no-store` 表示浏览器和中间缓存不得保存响应；`max-age=0` 明确响应立即过期，兼容检查工具和旧客户端。

宝塔在 Nginx `http` 层全局启用了 `proxy_cache`。只覆盖响应 `Cache-Control` 不会阻止 Nginx 在生成新响应之前复用已有缓存，因此 `/cn/`、`/jp/`、`/hk/` 及其子页面还必须同时设置基于请求路径的 `proxy_cache_bypass` 和 `proxy_no_cache`。该规则只作用于公开站点，CMS、API 和哈希静态资源继续沿用各自边界。

发布验收不再要求公开页面使用 `no-cache`，而是要求：

- 存在 `no-store`；
- 不得存在正数 `max-age` 或 `s-maxage`；
- HTML、RSC 连续请求均成功返回当前内容；
- `identity`、`gzip`、`br` 三种内容编码返回的 HTML/RSC 必须与当前 Next.js 进程一致，不能引用旧 release 的静态资源；
- 静态生成页面可以保留 ETag；运行时读取 CMS 数据的动态页面不强制要求 ETag，也不以缓存复用或 `304` 作为发布前提。

### 4.2 静态资源

`/_next/static/` 继续使用：

```text
Cache-Control: public, max-age=31536000, immutable
```

这些文件名包含内容哈希，新 release 会生成新的 URL；长期缓存不会导致新 HTML 使用旧资源。字体、图片等其他资源沿用现有已验证规则，不扩大本次变更范围。

### 4.3 CMS 与 API

`/cms/` 和 `/api/` 保留应用原有的 `private, no-store` 等响应头，不被公开页面规则覆盖。

### 4.4 历史缓存迁移边界

服务器新响应头无法影响浏览器已经保存且仍处于一年有效期内的旧 HTML。受影响设备需要通过一次带版本参数的页面或清除站点缓存取得新响应。服务器侧已经存在的宝塔代理缓存则不能归类为用户浏览器缓存，必须通过路径级代理缓存绕过立即失效；不能要求访问者自行刷新来规避服务器返回的旧版本。

本次不永久修改规范网址，也不在所有内部链接中长期附加版本参数。生产验收可使用一次性查询参数确认当前 release，正式站点仍保留规范路径。

## 5. CMS 三页签设计

### 5.1 页面结构

`/cms/leads/` 在告警和统计区域之后、详细筛选器之前显示三个明显页签：

1. **全部线索**：不设置 `kind`，展示所有线索。
2. **用户线索**：使用 `kind=TRIAL`。
3. **代理商线索**：使用 `kind=PARTNER`。

页签使用链接而不是仅靠客户端状态，URL 可复制、刷新和返回。当前页签有明确的选中状态、数量徽标和无障碍标识。

### 5.2 统计口径

服务端一次聚合返回：

- 全部线索总数；
- 用户线索总数；
- 代理商线索总数；
- 原有今日接收、待发送、已发送、已暂停和超时统计。

顶部五张运行状态卡继续表示全部线索的发送健康状态，避免切换页签时误判系统整体故障。三个页签上的数量只表示各类型总量。

### 5.3 筛选与链接行为

- 切换页签时保留当前 `status` 与 `overdue` 条件。
- 在页签内提交状态或预警筛选时保留当前 `kind`。
- “清除筛选”只清除状态和预警条件，保留当前页签。
- “全部线索”不输出空的 `kind` 参数。
- 非法 `kind` 继续按全部线索处理，不进入数据库查询。

### 5.4 列表与详情文案

- `TRIAL` 在列表和详情中统一显示为“用户线索”。
- `PARTNER` 统一显示为“代理商线索”。
- 联系人、脱敏电话、发送状态、等待时间、错误信息和操作保持现有逻辑。
- 两类线索继续共用 `/cms/leads/[id]/`，根据已有 `kind` 显示类型标题和对应表单字段。

### 5.5 空状态

空列表文案根据当前页签显示：

- 全部线索：没有符合条件的线索。
- 用户线索：没有符合条件的用户线索。
- 代理商线索：没有符合条件的代理商线索。

## 6. 数据流与兼容性

1. 公开表单仍分别调用 `/api/public/leads/trial/` 和 `/api/public/leads/partner/`。
2. 数据库继续把类型保存为 `TRIAL` 或 `PARTNER`，无需 schema 迁移。
3. 列表查询继续通过 `LeadFilters.kind` 过滤。
4. 统计查询增加按 `kind` 的聚合字段，不解密正文，不增加敏感信息暴露。
5. 现有线索发送 worker、外部接口映射、失败重试和飞书预警不变。

## 7. 错误处理

- 统计查询失败时沿用 CMS 服务端错误处理，不显示伪造的零数据。
- 非法筛选参数被忽略，不传入 SQL。
- 页签数量与列表使用同一数据库，避免分别读取不同数据源。
- Nginx 配置修改前继续备份，`nginx -t` 失败时自动恢复原配置。
- 发布脚本发现公开 HTML/RSC 缺少 `no-store` 时停止切换或触发回滚。
- standalone 服务必须在 PM2 自动重启后仍固定监听 `127.0.0.1:3000`，不能回退到系统 `HOSTNAME` 对应的 IPv6 地址；当前 release 可通过受控入口原地重启并执行完整生产健康检查。
- 预启动服务的后台 PID 必须对应实际 Node 进程，检查结束后完整退出，不能遗留占用 3100-3199 端口的进程。

## 8. 验收标准

### 8.1 自动验证

1. `TRIAL`、`PARTNER` 和全部类型数量统计正确。
2. 三个页签生成正确 URL，并保留状态和超时参数。
3. “清除筛选”保留当前类型页签。
4. 列表和详情使用“用户线索”“代理商线索”文案。
5. 公开 HTML 与 RSC 返回 `no-store, max-age=0`。
6. 实际引用的 CSS 和 JavaScript 继续返回一年 `immutable`。
7. CMS 与 API 继续返回私有无缓存头。

### 8.2 页面验收

1. 桌面端和移动端均能清晰看到三个页签及数量。
2. 切换页签后只显示对应类型线索。
3. 状态和预警筛选与页签组合正确。
4. 产品页使用一次性版本参数访问时显示新版对比方案。
5. 新取得的无参数公开页面响应不被浏览器保存；连续重新打开会再次请求服务器。

## 9. 已确认决策

- CMS 采用“全部线索｜用户线索｜代理商线索”三个明显页签。
- 公开 HTML 与 RSC 使用 `no-store, max-age=0`。
- 带哈希静态资源继续使用一年 immutable 缓存。
- 不修改现有线索数据库结构，不建设额外 CRM 功能。
