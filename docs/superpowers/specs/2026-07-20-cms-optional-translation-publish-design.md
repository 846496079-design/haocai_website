# CMS 非阻塞翻译发布设计

## 背景与根因

CMS 已经分别记录草稿完整度和 `CURRENT`、`STALE`、`NOT_TRANSLATED` 三种译稿状态，但现有发布门禁同时要求 CN、JP、HK 三种语言的标题、摘要、分类、封面和正文全部完整。前端也以同一三语完整度禁用“确认发布”。因此，翻译 API 未配置、调用失败或新媒体同事暂不翻译时，中文稿无法发布。

翻译 API 当前由 Next.js 服务端调用，密钥从 `CMS_TRANSLATION_API_KEY` 读取。生产发布脚本把 `/www/zhangdashi-deploy/shared/env/.env.local` 以软链接挂到每个 release，因此密钥应该进入该服务器共享环境文件，而不是网页、仓库、CNB 构建参数或任何 `NEXT_PUBLIC_` 变量。

## 已确认业务口径

- 中文稿达到现有必填标准、保存并完成当前版本预览后即可发布，翻译不是发布阻塞项。
- 未翻译或中文修改后译稿已过期时，中文站正常展示；日文站和香港站不展示该篇稿件。
- CMS 列表和编辑器继续明确标识“等待翻译”或“源稿有更新”。
- 后续完成翻译后，仍按“保存、预览、人工确认、再次发布”更新 JP/HK 公开站点。
- 已发布稿件正在编辑新草稿时，线上旧版本保持不变；只有新草稿正式发布后，外语站可见性才随新版本切换。

## 方案比较

### 方案 A：发布时固化外语公开状态（采用）

把“当前编辑稿的译稿状态”和“当前线上版本能否在外语站公开”分开。发布时根据当前草稿的三语完整度及译稿新鲜度，计算并写入 `published_locales_complete`。中文站始终读取已发布稿；JP/HK 仅读取该字段为真的已发布稿。

优点：发布切换是原子的；编辑新草稿不会提前影响旧线上版本；不需要清空或伪造日文、港文内容；后续翻译再发布即可恢复外语站展示。

### 方案 B：公开查询直接读取 `translation_status`

JP/HK 只展示 `translation_status = CURRENT` 的稿件。

缺点：`translation_status` 描述当前编辑稿。管理员修改已发布稿的中文后，它会立即变成 `STALE`，导致旧的线上日文和港文在新中文尚未发布时提前消失，不符合版本隔离要求。

### 方案 C：未翻译时在外语站回退显示中文

中文稿发布后，JP/HK 暂时展示中文内容。

缺点：会让用户误以为外语内容已经就绪，也不符合已确认的“外语站暂不展示”口径。

## 数据与兼容策略

在 `news_article` 增加 `published_locales_complete`：SQLite 使用 `INTEGER NOT NULL DEFAULT 0`，PostgreSQL 使用 `BOOLEAN NOT NULL DEFAULT FALSE`。

发布时按以下条件计算：

1. CN 必填项完整，否则仍拒绝发布。
2. JP、HK 必填项均完整。
3. `translation_status` 不是 `STALE`；首次由人工完整填写的外语稿即使未调用模型，也允许作为完整外语版本公开。
4. 条件 2、3 同时满足时写入真，否则写入假。

既有已发布数据在兼容迁移时按其已发布版本内容回填。完整的历史三语稿保持三站可见；缺少外语内容的历史稿只保留中文站展示。该回填只处理尚未初始化该字段的数据库，不覆盖后续发布形成的明确状态。

## 页面和接口行为

- 编辑器发布检查拆为“中文发布内容”和“外语公开状态”。
- “确认发布”不再因为 JP/HK 缺失而禁用，但仍受保存中、上传中、中文不完整、草稿未保存和当前版本未预览约束。
- 人工审核文案改为审核中文必填内容、封面、当前发布快照以及已填写的外语内容，不再声称必须审核完整三语。
- SQLite 与 PostgreSQL 的服务端发布函数执行相同校验，不能只放开前端按钮。
- `getPublishedArticles('jp' | 'hk')` 过滤未准备好的线上版本；`getPublishedArticle` 自然返回不存在，公开详情页按现有逻辑进入 404。
- 翻译失败只返回错误并保留草稿；不影响中文保存、预览和发布。

## API Key 生产配置

生产服务器只在 `/www/zhangdashi-deploy/shared/env/.env.local` 配置：

```env
CMS_TRANSLATION_API_URL=https://api.deepseek.com/chat/completions
CMS_TRANSLATION_API_KEY=<生产环境真实密钥>
CMS_TRANSLATION_MODEL=<供应商实际支持的模型标识>
```

文件权限保持 `0600`，修改后重启 PM2 的 `zds-website`。新媒体同事仍只登录 CMS 并点击“一键翻译”；浏览器请求本站 `/api/cms/news/:id/translate`，由服务器携带密钥访问模型供应商。浏览器源码、网络响应和公开页面都不能出现真实密钥。

## 验收标准

1. 只有中文内容的草稿可以保存、预览并发布，中文公开列表和详情可访问。
2. 同一篇未翻译稿不出现在 JP/HK 列表，直接访问 JP/HK 详情返回 404。
3. CMS 列表和编辑器显示未翻译或源稿已更新，不把它标记为已完成翻译。
4. 完成 JP/HK 翻译、保存、预览并重新发布后，三站均能访问。
5. 编辑一个已有三语线上稿的中文但尚未发布时，旧线上 JP/HK 仍然可访问。
6. 新中文草稿不重新翻译就发布后，中文站更新，JP/HK 隐藏该稿。
7. SQLite 和 PostgreSQL 两条存储路径执行相同规则，历史完整三语稿迁移后保持可见。
8. 服务器配置三项翻译变量并重启后，健康接口显示翻译已配置，CMS 能完成一次真实翻译；日志、页面和构建产物不包含 API Key。
