# CMS 公开新闻运行时数据设计

## 背景与根因

线上 CMS 已将稿件置顶状态写入共享数据库，CMS 已发布列表也能按置顶顺序读取。但中国站、日本站和香港站的首页、新闻列表及新闻详情在构建阶段被永久预渲染，构建进程读取的是临时初始化数据库，而不是服务器运行时共享数据库。

这会造成两个结果：

1. CMS 中发布、下架、置顶或调整顺序后，公开页面仍可能展示构建时的旧快照。
2. 每次部署都会重新带入构建阶段的旧新闻快照，即使此前运行时失效过页面缓存，也可能再次回到旧内容。

## 目标

- 三站所有依赖 CMS 新闻数据的公开页面，每次请求都从运行时共享数据库读取当前已发布版本。
- CMS 中的发布、下架、置顶和排序结果在下一次公开请求中直接生效。
- 公开 HTML 与 RSC 继续返回 `Cache-Control: no-store, max-age=0`。
- 带内容哈希的 CSS、JavaScript 和图片等静态资源继续使用长期不可变缓存。

## 方案比较

### 方案 A：CMS 依赖路由强制动态读取（采用）

将三站首页、新闻列表和新闻详情声明为动态路由。请求到达应用后读取共享数据库，再生成响应。

优点：数据口径直接、置顶立即生效、部署不会重新带回构建快照，符合“每次请求获取最新内容”的既定要求。当前新闻量较小，单次只读查询成本可控。

### 方案 B：短周期 ISR

保留静态页面，每 30 至 60 秒重新生成。

缺点：存在可见延迟；构建阶段仍可能读取临时数据库；需要额外处理部署后的首次快照，不符合已确认的实时读取要求。

### 方案 C：持久化 Next.js 页面缓存并修复失效链路

保留静态页面，引入共享缓存处理器，并让构建和运行时访问同一数据源。

缺点：部署和缓存基础设施复杂度明显增加，当前规模没有必要。

## 变更范围

仅调整下列九个公开路由：

- `/cn/`、`/jp/`、`/hk/`
- `/cn/news/`、`/jp/news/`、`/hk/news/`
- `/cn/news/[slug]/`、`/jp/news/[slug]/`、`/hk/news/[slug]/`

CMS、产品页、公司页等其他路由不改变渲染模式。新闻读取函数、数据库字段、置顶事务和排序 SQL 保持原有逻辑。

## 数据流

1. 管理员在 CMS 发布、下架、置顶或调整稿件顺序。
2. CMS 将状态写入服务器共享数据库，并保留原有审计记录和路径失效调用。
3. 用户访问三站首页、新闻列表或新闻详情。
4. 对应路由在运行时调用现有 `getPublishedArticles` 或 `getPublishedArticle`。
5. 数据库按置顶状态、置顶顺序、人工顺序和发布时间返回当前结果。
6. Nginx 对公开 HTML 与 RSC 响应增加 `no-store, max-age=0`，浏览器不保存旧页面。

## 异常处理

- 数据库不可用时沿用当前应用错误处理，不回退到构建期假数据，避免把旧内容误当成最新内容。
- 不修改 CMS 写入逻辑；已验证成功的置顶记录无需迁移。
- 不清理或重建共享数据库，避免影响现有稿件与管理员数据。

## 验收标准

1. 构建产物中九个 CMS 依赖路由不再标记为永久静态预渲染。
2. CMS 已发布列表的第一个置顶稿件，与三站新闻列表的第一篇一致。
3. 三站首页新闻区读取同一排序结果的前三篇。
4. 新闻详情读取共享数据库中的当前已发布版本。
5. 公开 HTML 和 RSC 返回 `no-store, max-age=0`，静态哈希资源仍为长期不可变缓存。
6. 线索管理提交随同部署上线，但本次修复不修改线索业务逻辑。
