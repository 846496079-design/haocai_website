# CMS 必填字段与发布校验引导设计

## 背景与根因

CMS 当前用 `isLocaleContentComplete` 判断中文标题、摘要、分类、封面、正文和服务端发布快照是否完整，但编辑器没有在字段标签上标识哪些内容必填。发布区只显示“缺少必填内容”，无法告诉用户具体缺少什么。

“确认发布”按钮还会在中文不完整、草稿未保存、当前版本未预览或尚未人工审核时直接禁用。禁用按钮没有点击反馈，用户只能自行猜测阻塞原因，形成无法继续的异常旅程。

截图中的当前稿件至少缺少“摘要”和“新闻封面”；发布日期、作者和标签不是中文发布必填项。

## 已确认业务口径

- 采用“必填标识、缺项汇总、字段错误和自动定位”组合方案。
- 点击“确认发布”时一次列出全部缺失的中文必填项，并滚动、聚焦第一个缺项。
- 发布按钮只有在保存、上传或翻译等操作正在执行时暂时禁用；其他发布条件不满足时仍允许点击并获得引导。
- 中文发布必填项是分类、新闻标题、摘要、新闻封面和正文内容。
- 发布日期、作者和标签保持选填，不新增业务门禁。
- 日文、繁体缺项只影响相应外语站公开，不阻塞中文发布。

## 方案比较

### 方案 A：只增加必填星号

在字段标签后显示 `*`，发布门禁保持不变。

优点是改动小；缺点是按钮仍不可点击，用户不知道当前究竟缺少哪些内容，也不知道保存、预览或审核是否仍未完成。

### 方案 B：必填标识、缺项汇总、字段错误和自动定位（采用）

在简体中文编辑页标出实际必填字段；点击发布时按照固定优先级检查当前状态，集中列出全部缺项，并把用户带到第一个需要处理的位置。

优点是用户始终能通过“确认发布”获得下一步，前端提示与服务端门禁可以共享同一字段定义，不会改变现有版本、预览和审核安全边界。

### 方案 C：独立发布向导弹窗

点击发布后进入多步骤弹窗，逐步处理内容、保存、预览和审核。

优点是流程强制；缺点是与当前单页工作台重复，步骤过重，且会增加编辑页与弹窗状态同步成本，因此不采用。

## 必填字段定义

### 用户可填写的中文必填项

| 字段标识 | 页面名称 | 完整条件 |
| --- | --- | --- |
| `category` | 分类 | 选择一个已启用分类 |
| `title` | 新闻标题 | 去除首尾空白后非空 |
| `summary` | 摘要 | 去除首尾空白后非空 |
| `cover` | 新闻封面 | 已上传并取得封面地址 |
| `body` | 正文内容 | 正文包含文字、图片或有效的外部排版块 |

`publicationHtml` 和 `contentHash` 是保存草稿时由服务端生成的发布快照字段，不作为用户可填写项显示。用户字段完整但发布快照缺失时，系统应引导重新保存草稿，而不是提示一个无法填写的内部字段。

### 页面标识

- 简体中文页显示：`分类 *`、`新闻标题 *`、`摘要 *`、`新闻封面 *`、`正文内容 *`。
- 编辑区顶部说明：`* 为中文发布必填；发布日期、作者、标签选填。`
- 星号使用可见文本和辅助技术可读说明，不只依赖红色表达必填。
- 日文和繁体页不把星号解释为中文发布门禁；外语完整度继续由“外语公开状态”单独说明。

## 点击“确认发布”的校验顺序

除正在保存、上传或翻译外，“确认发布”保持可点击。点击后按以下顺序只推进到当前最早的阻塞阶段：

1. **中文必填项缺失**：切换到简体中文页；汇总提示 `请补全：摘要、新闻封面`；所有缺项显示字段级错误；滚动并聚焦第一个缺项。
2. **草稿有未保存修改**：提示 `当前修改尚未保存，请先保存草稿。`，发布引导卡提供保存入口。
3. **当前保存版本未预览**：提示 `当前保存版本尚未预览，请先预览官网效果。`，发布引导卡提供预览入口。
4. **尚未人工审核**：提示 `请勾选人工审核确认。`，滚动并聚焦审核勾选框。
5. **全部条件满足**：调用现有发布接口并进入正式发布。

发布按钮在条件不满足时不伪装成成功，也不自动替用户保存、预览或勾选审核。系统负责说明和定位，用户仍显式完成每个有业务含义的动作。

## 字段错误与恢复

- 缺项字段增加 `aria-invalid` 和与字段关联的错误文案，例如 `请填写摘要`、`请选择新闻封面`、`请填写正文内容`。
- 用户补全某个字段后，该字段的错误状态立即消失；其他缺项继续保留。
- 再次点击发布时重新计算全部缺项，不沿用已经过时的错误列表。
- 新闻封面使用上传控件作为定位目标；正文使用富文本接收区外层作为定位目标。
- 自动滚动使用平滑定位，并把键盘焦点放到可编辑控件或可操作上传入口；不能只滚动但不提供焦点。

## 前后端一致性

- 在 CMS 类型层提供统一的中文可填写必填项检查函数，返回稳定字段标识和中文名称。
- 前端用该函数生成缺项汇总、字段级错误和第一个定位目标。
- SQLite 与 PostgreSQL 发布函数继续保留服务端门禁；缺少用户可填写必填项时返回具体字段列表和可读消息。
- 用户字段完整但服务端发布快照缺失时，服务端返回“请重新保存草稿”类错误，不暴露 `publicationHtml` 或 `contentHash` 等内部字段名。
- 发布接口保留结构化错误码和字段列表，前端优先使用结构化数据，未知错误仍显示服务端可读消息。
- 外语完整度、`translation_status` 和 `published_locales_complete` 的现有规则不变。

## 异常处理

- 发布请求期间按钮暂时禁用，避免重复提交；请求结束后恢复。
- 服务端返回缺项时，即使前端之前判断完整，也按服务端字段列表更新提示并定位，处理多窗口或旧页面状态差异。
- 稿件状态变化、登录失效或网络失败继续显示明确错误，不把这些错误误报成必填缺项。
- 聚焦目标暂时不可用时，至少保留缺项汇总和字段错误，不能吞掉提示。

## 验收标准

1. 简体中文页仅在分类、新闻标题、摘要、新闻封面和正文内容上显示必填星号，并明确说明发布日期、作者、标签选填。
2. 中文缺少摘要和封面时，“确认发布”仍可点击；点击后一次显示 `请补全：摘要、新闻封面`，两个字段均标红，并聚焦摘要。
3. 补全摘要后，摘要错误立即消失；再次点击发布只提示新闻封面并定位上传入口。
4. 中文必填项完整但草稿未保存时，点击发布引导保存；不会直接调用正式发布接口。
5. 草稿已保存但未预览时，点击发布引导预览；不会自动标记预览完成。
6. 当前版本已预览但未审核时，点击发布提示并聚焦人工审核勾选框。
7. 全部条件满足后正常发布；JP/HK 不完整仍不阻塞中文站发布。
8. SQLite 和 PostgreSQL 服务端都返回具体中文缺项；发布快照异常提示重新保存，不泄露内部字段名称。
9. 翻译、上传、保存和发布执行期间防重复操作，现有翻译整页锁定、离页提醒和版本隔离没有回归。
10. 页面中文无乱码，键盘和辅助技术能够识别必填、错误及当前焦点。
