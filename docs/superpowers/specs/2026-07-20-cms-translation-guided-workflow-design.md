# CMS 翻译锁定与发布引导设计

## 背景与根因

生产环境已经能够调用翻译服务，但当前编辑器在翻译过程中复用 `saving` 状态，只禁用了部分按钮和富文本编辑器。语言切换、普通输入项、审核勾选等控件仍可能被操作，用户可能在请求尚未完成时修改页面，造成对最终保存内容的误判。

翻译接口会把生成的日文和繁体内容写入当前草稿，并使之前的发布预览失效。现有页面只显示“需要重新预览”并禁用“确认发布”，没有提供直接的下一步入口。生产页面还未部署非阻塞翻译发布版本，因此仍使用旧的“三语内容完整”门禁和“待补全或保存”文案，把已完成翻译、外语缺项、中文缺项、未保存等不同状态混为一谈。

本设计补充翻译过程锁定和发布旅程引导；中文可独立发布、未就绪外语稿不公开的业务规则继续以 `2026-07-20-cms-optional-translation-publish-design.md` 为准。

## 已确认业务口径

- 翻译请求执行期间锁定整个编辑工作区，避免编辑、切换语言、保存、预览、审核或发布等误操作。
- 翻译完成后仍要求用户查看当前保存版本的官网预览，不自动把未经检查的内容标记为已预览。
- 系统必须明确告诉用户当前阻塞原因，并提供可直接执行的下一步操作，不能只留下禁用按钮。
- 日文或繁体内容不完整时不阻塞中文发布，但要准确标识外语稿尚不能随本次发布上线。
- 翻译失败保留翻译前的已保存草稿，解除页面锁定，并向用户显示可重试的错误信息。

## 方案比较

### 方案 A：翻译结束后只显示成功提示

翻译成功后显示一次提示，用户自行寻找预览按钮。

优点是改动小；缺点是提示容易被忽略，发布按钮仍然处于禁用状态，无法解决用户不知道下一步怎么走的问题。

### 方案 B：状态锁定加持续的下一步引导卡（采用）

翻译期间显示全页面处理遮罩；翻译成功并重新载入服务端草稿后，在发布检查区显示持续的“下一步”引导卡。引导卡根据当前状态提供“保存草稿”“预览官网效果”“勾选人工审核”或“确认发布”等明确动作。

优点是保留人工预览门禁，同时让禁用原因和下一步入口保持在同一区域；刷新后的数据来自服务端，避免用浏览器中的旧状态猜测翻译是否保存成功。

### 方案 C：翻译成功后自动打开并自动记录预览

翻译成功后直接打开预览窗口并把版本标记为已预览。

缺点是异步请求完成后打开新窗口可能被浏览器拦截，而且“已打开”不等于用户已经检查。自动记录会削弱现有发布审核门禁，因此不采用。

## 页面状态与交互

### 翻译处理中

编辑器新增独立的 `translating` 状态，不再用 `saving` 同时表达保存和模型翻译。

- 点击“保存并翻译日文、繁体”后，先保存当前草稿；保存失败时不发起翻译。
- 翻译请求开始后显示覆盖整个编辑工作区的遮罩，文案为“正在翻译并保存日文、繁体内容，请勿关闭页面”。
- 遮罩显示明确的处理中状态，拦截页面内点击和键盘输入；底层表单控件及操作按钮同步禁用。
- 翻译期间注册离开页面提醒，用户刷新、关闭标签页或跳转时由浏览器提示可能中断当前操作。请求结束后立即移除提醒。
- 不提供“取消翻译”按钮，因为浏览器取消请求不能保证服务端和模型供应商同时终止，容易造成用户认为已取消但服务端随后完成保存的状态歧义。

### 翻译成功

- 翻译接口继续在服务端保存生成结果并记录译稿状态。
- 页面重新读取服务端草稿，确保版本号、更新时间、发布快照和译稿状态一致。
- 页面显示“翻译已完成并保存。下一步：预览官网效果”。
- 引导卡直接提供“预览官网效果”主按钮；用户不需要再从页面其他位置寻找入口。
- 翻译改变了当前草稿，因此此前预览保持失效，“确认发布”在重新预览前继续禁用。

### 预览完成

- 用户点击引导卡中的“预览官网效果”后，系统先记录当前服务端版本已预览，再打开预览页。
- 编辑页提示“预览已打开，请检查内容和封面；确认无误后返回本页勾选人工审核”。
- 当前版本已预览但尚未勾选审核时，引导卡的下一步改为“勾选人工审核”。
- 人工审核已勾选且其他发布条件满足时，引导卡显示“检查已完成，可以确认发布”，发布按钮启用。

### 翻译失败

- 解除全页面锁定和离开页面提醒。
- 页面显示翻译接口返回的可读错误；没有可读错误时使用“翻译失败，请检查网络或翻译服务后重试”。
- 已保存的中文和翻译前外语内容保持不变，用户可以重试翻译，也可以按非阻塞规则继续保存、预览并只发布中文站。

## 发布检查与文案

发布检查继续拆分为：

1. `中文发布内容`：只描述中文必填项和当前草稿保存状态。
2. `外语公开状态`：描述日文、繁体是否能随本次版本上线；外语未就绪不禁用中文发布。
3. `译稿状态`：区分已基于当前中文生成、中文修改后译稿过期、尚未生成。
4. `当前版本预览`：区分已完成和需要重新预览。
5. `图片上传`：区分进行中和已完成。

页面不再使用“三语内容：待补全或保存”这一合并状态。发布按钮禁用时，引导卡按以下优先级只给出一个当前最关键动作：

1. 正在翻译或上传：等待当前操作完成。
2. 中文必填内容不完整：补全中文必填内容。
3. 草稿有未保存修改：保存草稿。
4. 当前版本未预览：预览官网效果。
5. 尚未人工确认：勾选人工审核。
6. 全部完成：确认发布。

## 数据与接口边界

- 不新增数据库字段，继续使用现有 `previewed_at`、`translation_status` 和 `published_locales_complete`。
- 翻译成功的服务端写入顺序保持为：更新草稿、标记译稿为 `CURRENT`、完成翻译任务记录。
- 前端不自行伪造“已保存”或“已预览”状态；翻译后通过重新读取服务端数据恢复编辑器状态。
- API Key 仍只由服务端环境变量读取，不进入前端状态、查询参数、提示文案或日志。
- 中文独立发布和 JP/HK 公开过滤继续由前后端双重门禁保证，不因本次交互调整而改变。

## 可访问性与防误操作

- 处理遮罩使用可被辅助技术识别的忙碌状态和即时状态播报。
- 遮罩出现时把焦点移动到处理提示，结束后把焦点移动到成功引导或错误提示。
- 禁用控件同时保留可见的禁用原因，不以颜色作为唯一状态表达。
- 页面文案使用中文 UTF-8，不使用图标字符或 emoji 表示关键状态。

## 验收标准

1. 翻译开始后，编辑区、语言切换、保存、复制、预览、审核和发布均不可操作，页面显示明确的处理中遮罩。
2. 翻译期间尝试刷新、关闭或跳转时出现浏览器离开提醒；翻译完成或失败后不再提醒。
3. 翻译成功后，日文和繁体内容已写入服务端草稿，页面显示“翻译已完成并保存”及可点击的“预览官网效果”。
4. 翻译成功不会自动记录预览；重新预览前“确认发布”保持禁用，并明确说明下一步是预览。
5. 预览打开后，编辑页提示用户检查预览并返回勾选人工审核；完成审核后发布按钮启用。
6. 翻译失败后页面解除锁定、保留翻译前已保存内容并显示重试提示，中文稿仍可独立走保存、预览和发布流程。
7. 生产部署非阻塞版本后，页面不再显示“三语内容：待补全或保存”；JP/HK 缺项只影响外语公开状态，不阻塞中文发布。
8. SQLite 和 PostgreSQL 运行路径的发布规则不变，现有翻译任务审计、预览门禁和版本隔离验证继续通过。
9. 中文界面无乱码，不出现内部开发提示或 API Key。
